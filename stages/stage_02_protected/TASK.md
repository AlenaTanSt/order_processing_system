## Цель этапа

Перевести конвейер обработки заказов из `stage01` на **параллельную обработку с разделяемыми структурами данных**, обеспечив:

* корректность результата;
* отсутствие гонок данных;
* детерминированность конечного состояния.

Ключевое отличие от `stage01`: теперь у рабочих потоков есть **общий источник работы** (очередь), и без синхронизации корректность невозможна.

## Общие требования

* Обработка должна использовать **несколько потоков** (через `std::thread`) при наличии более одного заказа.
* Итоговое поведение должно оставаться корректным:

  * каждый заказ завершается статусом `Delivered`;
  * порядок заказов в результате соответствует порядку подачи;
  * метрики корректны.
* Использовать **только стандартную библиотеку C++**.
* Все файлы должны располагаться **строго в папке `solution/`**.
* Тесты должны стабильно проходить при многократных запусках.

## Что необходимо реализовать

### 1) Добавить файл `solution/include/ops/blocking_queue.hpp`

Файл должен реализовывать **потокобезопасную блокирующую очередь** с поддержкой закрытия.

Очередь используется для распределения работ между потоками и должна обеспечивать корректное завершение воркеров без busy-wait.

```cpp
template <class T>
class BlockingQueue {
public:
    BlockingQueue() = default;

    BlockingQueue(const BlockingQueue&) = delete;
    BlockingQueue& operator=(const BlockingQueue&) = delete;

    // Добавляет элемент в очередь.
    // Возвращает false, если очередь закрыта и элемент не принят.
    bool push(T value);

    // Блокируется, пока:
    // - в очереди не появится элемент, или
    // - очередь не будет закрыта.
    // Возвращает std::nullopt, если очередь закрыта и пуста.
    std::optional<T> pop_blocking();

    // Закрывает очередь и будит всех ожидающих потоков.
    void close();

private:
    std::mutex m_;
    std::condition_variable cv_;
    std::queue<T> q_;
    bool closed_ = false;
};
```

#### Требования к поведению `BlockingQueue`

* `push()`:

  * добавляет элемент, если очередь не закрыта;
  * после `close()` всегда возвращает `false`.
* `pop_blocking()`:

  * не использует busy-wait;
  * блокируется через `condition_variable`;
  * возвращает `std::nullopt`, если очередь закрыта и пуста.
* `close()`:

  * переводит очередь в состояние "закрыта";
  * будит **всех** ожидающих потоков.

Эта очередь не используется как входная очередь Pipeline. Она применяется исключительно для распределения работ между потоками.

## Требования к `submit()`

* работает аналогично `stage01`;
* увеличивает `accepted_count`;
* добавляет заказ во входную очередь `queue_`.

## Требования к `process_all()`

### 1) Подготовка батча (однопоточно)

* извлечь все заказы из `queue_` в локальный `std::vector<Order> orders`;
* если `orders` пуст, функция должна **немедленно завершаться**:

  * без создания потоков;
  * без изменения метрик;
  * без изменения `delivered_`.

### 2) Выбор количества потоков

Определить `workers`:

* `workers = std::thread::hardware_concurrency();`
* если вернулось 0, считать `workers = 2`;
* `workers <= orders.size()`;
* минимум 1.

Дополнительно:

* если `orders.size() == 1`, допускается 1 поток;
* если `orders.size() >= 2`, должно быть создано **не менее двух потоков**.

### 3) Распределение работ через очередь

* создать `BlockingQueue<std::size_t>` для индексов заказов;
* добавить в очередь индексы `0 .. orders.size() - 1`;
* вызвать `close()` после заполнения очереди;
* создать `workers` потоков.

Каждый поток в цикле:

* вызывает `pop_blocking()`;
* если получен `std::nullopt`, завершает работу;
* иначе обрабатывает **один заказ** `orders[idx]`, полностью прогоняя его по стадиям:

`Accepted -> Prepared -> Packed -> Delivered`.

#### Важно

Внутри потоков **запрещено**:

* изменять `metrics_`;
* писать в `delivered_`;
* обращаться к входной `queue_`.

Поток может модифицировать **только** `orders[idx]`.

### 4) Детерминированный порядок результата

* после `join()` всех потоков:

  * пройти `orders` **слева направо**;
  * добавить заказы в `delivered_` в этом порядке.

Это гарантирует соответствие порядку подачи.

### 5) Метрики (однопоточно)

После завершения всех потоков:

* `processed_count += batch_size;`
* `delivered_count += batch_size;`
* `total_processing_time += сумма(delivered_time - accepted_time)` по всем заказам батча.

Метрики являются **накопительными** между вызовами `process_all()`.

### 6) Безопасное владение потоками

* запрещено выходить из `process_all()` с `joinable` потоками;
* `detach` запрещен;
* каждый созданный поток должен быть `join()` до выхода из функции.

## Инварианты корректности

После обработки каждого заказа должны выполняться:

* `accepted_time <= prepared_time`
* `prepared_time <= packed_time`
* `packed_time <= delivered_time`

## Критерии приемки

Этап считается выполненным, если:

* проект собирается без ошибок;
* все тесты проходят стабильно;
* при `orders.size() >= 2` реально используются минимум два потока;
* отсутствует busy-wait;
* не используются `atomic`, `future`, `async`, `detach`;
* все заказы имеют статус `Delivered`;
* порядок `delivered_orders()` соответствует порядку подачи;
* временные метки стадий не убывают;
* метрики корректны и обновляются только после `join()`.
