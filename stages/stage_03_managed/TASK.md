## Цель этапа

Перевести обработку заказов из `stage_02` на **настоящий конвейер по стадиям**, где:

* каждая стадия обработки выполняется в отдельном наборе потоков;
* между стадиями используются **потокобезопасные блокирующие очереди**;
* система корректно завершается без busy-wait и без утечек потоков;
* конечный результат детерминирован (порядок `delivered_orders()` совпадает с порядком подачи).

Ключевое отличие от `stage_02`:

* в `stage_02` потоки берут задания из одной очереди и целиком обрабатывают заказ;
* в `stage_03` заказ проходит **последовательно через несколько очередей**, каждая соответствует стадии.

## Общие требования

* При наличии более одного заказа должны использоваться **несколько потоков** (`std::thread`).
* Итоговое поведение должно оставаться корректным:

  * каждый заказ завершается статусом `Delivered`;
  * порядок заказов в результате соответствует порядку подачи;
  * метрики корректны и накопительны.
* Использовать только стандартную библиотеку C++.
* Все файлы строго в `solution/`.
* Тесты должны стабильно проходить при многократных запусках.

## Что необходимо реализовать

## 1) Очереди стадий

Использовать `BlockingQueue<T>` как основу для связи стадий. Потребуется несколько очередей:

* `q_prepare`: очередь заказов на стадию `Prepared`
* `q_pack`: очередь заказов на стадию `Packed`
* `q_deliver`: очередь заказов на стадию `Delivered`

Тип очередей: `BlockingQueue<Order>` (перемещение заказа допускается и предпочтительно).

## 2) Pipeline: конвейерное выполнение

### Публичный интерфейс (не меняется)

* `void submit(Order order);`
* `void process_all();`
* `const Metrics& metrics() const;`
* `const std::vector<Order>& delivered_orders() const;`

### Требования к submit()

* работает как раньше:

  * увеличивает `accepted_count`;
  * кладет заказ во входную неблокирующую очередь `Pipeline::queue_`.

## 3) Требования к process_all()

### 3.1 Подготовка батча (однопоточно)

* извлечь все заказы из входной неблокирующей очереди `queue_` в `std::vector<Order> orders`;
* если батч пуст:

  * немедленно выйти;
  * не создавать потоки;
  * не менять `metrics_` и `delivered_`.

### 3.2 Запуск стадий (многопоточно)

Внутри `process_all()` создать очереди стадий и запустить обработчиков стадий:

#### Стадия A: Prepare

* читает из `q_prepare`
* переводит заказ: `Accepted -> Prepared`
* пишет заказ в `q_pack`

#### Стадия B: Pack

* читает из `q_pack`
* переводит заказ: `Prepared -> Packed`
* пишет заказ в `q_deliver`

#### Стадия C: Deliver

* читает из `q_deliver`
* переводит заказ: `Packed -> Delivered`
* сохраняет результат в локальный контейнер `done` (внутри `process_all()`), но **не пишет в `delivered_` из потоков**

Количество потоков на каждой стадии выбирается аналогично `stage_02`:

* `workers = std::thread::hardware_concurrency();` если 0 -> 2
* `workers <= batch_size`
* минимум 1
* если `batch_size >= 2` — суммарно должно быть создано не менее 2 потоков (можно распределить по стадиям, но проще: по 1 потоку на каждую стадию, и при batch_size>=2 увеличить хотя бы одну стадию)

Рекомендуемая схема:

* `prepare_workers = max(1, workers/3)`
* `pack_workers = max(1, workers/3)`
* `deliver_workers = max(1, workers - prepare_workers - pack_workers)`
* при маленьких batch корректировать так, чтобы не создавать лишних потоков.

### 3.3 Закрытие очередей и завершение потоков

* после заполнения `q_prepare` всеми заказами батча вызвать `q_prepare.close()`;
* каждый worker стадии, получив `nullopt`, завершает работу;
* после завершения всех worker стадии `Prepare` (join) закрыть `q_pack`;
* после завершения всех worker стадии `Pack` (join) закрыть `q_deliver`;
* после завершения всех worker стадии `Deliver` (join) стадийные потоки должны быть полностью завершены.

Запрещено выходить из `process_all()` с joinable потоками.

### 3.4 Детерминированный порядок результата

Для сохранения порядка подачи:

* каждому заказу присвоить индекс `i` (позиция в батче);
* переносить между очередями не просто `Order`, а структуру:

```cpp
struct IndexedOrder {
    std::size_t index;
    Order order;
};
```

* стадийные очереди имеют тип `BlockingQueue<IndexedOrder>`;
* финальная стадия сохраняет результаты в `std::vector<Order> done(batch_size)` по `index`;
* после join всех потоков:

  * слева направо перенести `done` в `delivered_`.

Это гарантирует детерминированность и стабильность тестов.

### 3.5 Метрики (однопоточно)

После завершения всех стадий:

* `processed_count += batch_size;`
* `delivered_count += batch_size;`
* `total_processing_time += сумма(delivered_time - accepted_time)` по всем заказам батча.

Метрики накопительны между вызовами `process_all()`.

## Инварианты корректности

После обработки каждого заказа должны выполняться:

* `accepted_time <= prepared_time`
* `prepared_time <= packed_time`
* `packed_time <= delivered_time`

## Критерии приемки

Этап считается выполненным, если:

* проект собирается без ошибок;
* тесты проходят стабильно при многократных запусках;
* при `batch_size >= 2` используется реальная параллельность (минимум 2 потока);
* отсутствует busy-wait;
* не используются `atomic`, `future`, `async`, `detach`;
* все заказы завершены статусом `Delivered`;
* `delivered_orders()` соответствует порядку подачи;
* метрики корректны и обновляются только после завершения потоков.
