## Цель этапа

Реализовать последовательную (однопоточную) систему обработки заказов.
На этом этапе запрещено использовать потоки, мьютексы, атомики и любые средства конкурентности.

Этот этап является базовой точкой отсчета для всего курса. В дальнейшем именно этот код будет последовательно дорабатываться и распараллеливаться.

## Общие требования

- Вся обработка выполняется в одном потоке. 
- Код должен быть:

	- корректным,

	- детерминированным,

	- читаемым.

- Использовать только стандартную библиотеку C++.

- Все файлы должны располагаться только в папке solution/.

- Тесты должны успешно проходить.

## Модель предметной области
### Заказ (Order)

Заказ проходит следующие стадии обработки строго по порядку:

- **Accepted** – заказ принят

- **Prepared** – заказ подготовлен

- **Packed** – заказ упакован

- **Delivered** – заказ доставлен (финальное состояние)

### Каждый заказ имеет:

- уникальный идентификатор (```std::uint64_t```),

- текущий статус,

- временные метки прохождения стадий (```std::chrono::steady_clock::time_point```) - они будут использоваться в следующих этапах:
	- ```std::chrono::steady_clock::time_point accepted_time```;
	  														
	- ```std::chrono::steady_clock::time_point prepared_time```;
	  														
	- ```std::chrono::steady_clock::time_point packed_time```;
	  														
	- ```std::chrono::steady_clock::time_point delivered_time```;

## Что необходимо реализовать
### 1. order.hpp

- Перечисление **OrderStatus** со всеми стадиями;

- Алиас **OrderId = std::uint64_t**;

#### Реализовать класс Order:

```
class Order {
public:
    explicit Order(OrderId id_); - конструктор с параметрами

    void advance_to(OrderStatus new_status); - возможность смены состояния

public:
    OrderId id; - уникальный идентификатор
    OrderStatus status; - текущее состояние заказа

    std::chrono::steady_clock::time_point accepted_time;
    std::chrono::steady_clock::time_point prepared_time;
    std::chrono::steady_clock::time_point packed_time;
    std::chrono::steady_clock::time_point delivered_time;
};

```

#### Требования:

- идентификаторы должны быть уникальными в рамках запуска программы;

- начальное состояние — *Accepted*.

### 2. queue.hpp

Реализовать простую очередь заказов без потокобезопасности в классе **OrderQueue**.

#### Требуемый интерфейс (минимум):

- ```push(const Order&)``` или ```push(Order&&)``` - добавляет заказ в очередь

- ```pop()``` — извлекает заказ из очереди

- ```empty()``` - проверяет очередь на пустоту

Можно использовать ```std::queue``` или ```std::deque```.

### 3. pipeline.hpp / pipeline.cpp

Реализовать класс **Pipeline**, который управляет обработкой заказов.

#### Обязанности Pipeline:

- Принимать новые заказы.

- Последовательно проводить каждый заказ через все стадии.

- Завершать обработку заказов в состоянии *Delivered*.

- Обновлять метрики обработки заказов.

#### Минимальный интерфейс:
```
class Pipeline {
public:
    void submit(Order order); — добавить заказ в систему.
    void process_all(); — обработать все заказы до конца.

    const Metrics& metrics() const; — получить текущие метрики обработки.
    const std::vector<Order>& delivered_orders() const; - возвращает список завершенных заказов.

private:
    OrderQueue queue_;
    Metrics metrics_;
    std::vector<Order> delivered_;
};

```

#### Требования к process_all():

- заказы обрабатываются строго последовательно;

- каждая стадия применяется ко всем заказам;

- порядок заказов сохраняется.

### 4. metrics.hpp

Реализовать простую систему метрик в формате структуры **Metrics**.

#### Минимум:

- счетчик принятых заказов (```std::size_t accepted_count```);
- счетчик доставленных заказов (```std::size_t delivered_count```);
- счетчик обработанных заказов (```std::size_t processed_count```);
- общее время обработки заказов (```std::chrono::steady_clock::duration total_processing_time```) : ```delivered_time - accepted_time```;

Счетчики должны корректно обновляться в процессе обработки.

## Критерии приемки

Этап считается выполненным, если:

- Проект собирается без ошибок.

- Все тесты проходят успешно.

- Все заказы доходят до состояния *Delivered*.

- Код не использует средства конкурентности.

- Реализация читаема и не содержит лишней логики.
